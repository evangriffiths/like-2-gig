# Like2Gig

Find upcoming gigs near you for artists in your Spotify 'Liked Songs' playlist.

## Features

- **Artists** ‚Äî syncs your Spotify liked songs, shows all artists with song counts and dates
- **Gigs** ‚Äî scrapes Songkick for upcoming gigs, filterable by location and date range
- **Notifications** ‚Äî set up email alerts for new gigs in specific locations
- **Multi-user** ‚Äî each Spotify login gets their own data; gig scrape results are shared
- **Background sync** ‚Äî artist + gig syncing runs in the background with live progress
- **Daily cron** ‚Äî automated sync at 4am UTC via cron job

## Local Development

### Prerequisites

- Node.js 22+
- pnpm

### Setup

1. Create a Spotify app at <https://developer.spotify.com/dashboard>
   - Add `http://127.0.0.1:5173/auth/callback` as a redirect URI

2. Create a `.env` file in the project root:

   ```bash
   CLIENT_ID=your_spotify_client_id
   CLIENT_SECRET=your_spotify_client_secret
   RESEND_API_KEY=your_resend_api_key  # optional, for email notifications
   ```

3. Install dependencies and start:

   ```bash
   pnpm install
   pnpm dev
   ```

The client runs at `http://127.0.0.1:5173` and the server at `http://127.0.0.1:3001`.

## Architecture

```
Client (React + Vite)         Server (Express + SQLite)
‚îú‚îÄ‚îÄ /artists                  ‚îú‚îÄ‚îÄ /api/liked-artists (cached)
‚îú‚îÄ‚îÄ /gigs                     ‚îú‚îÄ‚îÄ /api/gigs (read-only)
‚îú‚îÄ‚îÄ /notifications            ‚îú‚îÄ‚îÄ /api/notifications (CRUD)
                              ‚îú‚îÄ‚îÄ /api/sync (background job)
                              ‚îú‚îÄ‚îÄ /api/sync-status (polling)
                              ‚îú‚îÄ‚îÄ /api/geocode (Nominatim proxy)
                              ‚îú‚îÄ‚îÄ /api/cron/sync (all users)
                              ‚îî‚îÄ‚îÄ /auth/* (Spotify OAuth)
```

**Sync flow:**

1. `POST /api/sync` kicks off a background job, returns immediately
2. Phase 1: fetch liked songs from Spotify, cache artists per user
3. Phase 2: scrape Songkick for stale artists' gigs (shared across users)
4. Post-sync: detect new gigs, evaluate notification rules, send emails
5. Client polls `GET /api/sync-status` for progress

**Data model:**

- `users` ‚Äî Spotify user ID, email, refresh token (per user)
- `liked_artists` / `liked_songs` ‚Äî per user
- `artist_gigs` / `gigs` ‚Äî shared (gig data is per artist, reused across users)
- `notification_rules` / `notification_log` ‚Äî per user
- `sync_jobs` ‚Äî per user, tracks background sync progress

## Deployment

Deployed on a Hetzner VPS (CX22, Ubuntu) at <https://like2gig.evangriffiths.org>.

### Server config

All server configuration lives in `deploy/`:

| File | Purpose | Deployed to |
|------|---------|-------------|
| `deploy/setup.sh` | One-time server bootstrap | Run manually on fresh VPS |
| `deploy/like2gig.service` | systemd service (Node app) | `/etc/systemd/system/` (via deploy hook) |
| `deploy/Caddyfile` | Reverse proxy + SSL + basic auth | `/etc/caddy/Caddyfile` (manual) |
| `deploy/post-receive` | Git deploy hook | `/opt/like2gig.git/hooks/` |
| `deploy/crontab` | Daily sync at 4am UTC | `crontab -e` on server |

### Fresh server setup

1. **Provision a VPS** on Hetzner with primary IPv4, add your SSH key.

2. **Run the setup script:**

   ```bash
   ssh root@<your-ip>
   # Copy deploy/setup.sh to the server and run it:
   bash setup.sh
   ```

3. **Edit `/opt/like2gig/.env`** with your credentials (see env vars below).

4. **Set up DNS** ‚Äî add an A record pointing your subdomain to the server IP.

5. **Configure Caddy** ‚Äî copy `deploy/Caddyfile` to `/etc/caddy/Caddyfile`, set the password hash with `caddy hash-password --plaintext <password>`, then `systemctl restart caddy`.

6. **Add the redirect URI** in the Spotify Developer Dashboard: `https://your-domain/auth/callback`

7. **First deploy** from your local machine: `git push deploy main`

8. **Allow native builds** (first time only):

   ```bash
   # On server, in /opt/like2gig:
   pnpm rebuild better-sqlite3
   ```

9. **Set up cron** ‚Äî edit `deploy/crontab` with your `CRON_SECRET`, then on the server: `crontab deploy/crontab`

### Environment variables

`/opt/like2gig/.env` on the server:

```bash
CLIENT_ID=your_spotify_client_id
CLIENT_SECRET=your_spotify_client_secret
SESSION_SECRET=<auto-generated by setup.sh>
CRON_SECRET=<auto-generated by setup.sh>
BASE_URL=https://like2gig.evangriffiths.org
RESEND_API_KEY=your_resend_api_key
```

### Email notifications (Resend)

1. Sign up at <https://resend.com> (free tier: 100 emails/day)
2. Add your domain in Resend and add the DNS records it gives you (DKIM, SPF MX, SPF TXT)
3. Wait for domain verification
4. Add `RESEND_API_KEY` to `.env` on the server

### Deploying changes

Pushing to `main` on GitHub automatically triggers a deploy via GitHub Actions. The workflow SSHs into the server and pushes to the bare git repo, which runs the post-receive hook.

**One-time setup for CI:**

```bash
# Add your SSH private key as a GitHub secret
gh secret set DEPLOY_SSH_KEY < ~/.ssh/id_ed25519
```

Then just push to GitHub as normal:

```bash
git push origin main
```

**Manual deploy** (bypasses GitHub):

```bash
# One-time: add the deploy remote
git remote add deploy root@<your-ip>:/opt/like2gig.git

# Deploy directly
git push deploy main
```
